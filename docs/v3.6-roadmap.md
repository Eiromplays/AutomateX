# AutomateX v3.6 roadmap — the multi-tenant / exposed tier

**Status:** planning. v3.5 shipped the first half of the ops/hardening track (metrics + failure
alerting in v3.5.0, idempotency keys in v3.5.1). v3.6 finishes it: the three items that matter once an
instance is **exposed to the internet or shared between tenants**.

## TL;DR

Three themes. Recommended order is by **leverage and risk** — foundational + low-risk first, the big
isolation rewrite last:

1. **Audit log + instance-admin role.** Who created/changed/ran/deleted what, and a role above
   workspace-owner for instance operators. Foundational, moderate size, low blast radius. **Start here.**
2. **Per-tenant DEKs + secret rotation.** Per-tenant data-encryption keys under the existing envelope
   scheme, plus a rotation path. Self-contained security depth; the `v1:`/`v2:` ciphertext prefix is
   already in place.
3. **Out-of-proc plugin sandboxing.** The real isolation boundary — each plugin in its own process.
   Largest single item, highest risk; **could be its own release or slip to v4.**

> You listed these as sandboxing → DEKs → audit log. I'd flip it: lead with the audit log (cheap,
> high-value, no risk) and end with sandboxing (expensive, risky), so the hard rewrite lands on a
> codebase that already has audit + per-tenant keys to lean on. Confirm the first milestone before I
> build — see the bottom.

## v3.6a — Audit log + instance-admin role

Today actions on workflows/connections/triggers leave no durable trail, and the top role is
workspace-owner — there's no "instance operator" who can see across workspaces.

### Audit log

An append-only record of mutations: actor, action, target (type + id), workspace, timestamp, and a
small diff/summary. Read API + a UI view, filterable by actor/target/workspace.

- **Seam:** mutations funnel through FastEndpoints handlers; `WorkspaceAccess.AuthorizeAsync` already
  resolves the actor + workspace on every protected write. Capture there (a small `IAuditSink` called
  from handlers, or an endpoint post-processor) → an `AuditEntry` table. Engine-side run/resume/retry
  events can reuse the existing `IListenFor<>` bus (like the metrics listener).
- **Tests-first:** a create/update/delete writes one entry with the right actor + target; reads are
  workspace-scoped (an operator sees all; a member sees their workspace).

### Instance-admin role

A role above `WorkspaceRole.Owner` for operators: see/manage all workspaces, read the global audit
log, manage instance settings. Gate via config (`Auth__InstanceAdmins` = a set of subjects/keys) so
it works in the open/API-key/OIDC tiers.

- **Seam:** extend `WorkspaceAccess` with an `IsInstanceAdmin` check; instance-admin endpoints sit
  outside the per-workspace gate.
- **Risk:** privilege escalation — over-test the authorization boundary; an instance-admin grant must
  be explicit (config), never inferred.

## v3.6b — Per-tenant DEKs + secret rotation

Today connection secrets are AES-256-GCM under one instance key (`Encryption__Key`, never stored),
with a `v1:`/`v2:` ciphertext prefix already carrying the envelope scheme. v3.6b makes the
data-encryption key **per tenant** and adds rotation.

- **Seam:** `SecretCipher` is the one chokepoint for encrypt/decrypt. Introduce a per-workspace DEK,
  itself wrapped by the instance KEK (`Encryption__Key`); the prefix already lets old and new
  ciphertexts coexist, so decryption stays backward-compatible while new writes use the per-tenant DEK.
- **Rotation:** a re-encrypt pass (KEK rotation re-wraps DEKs; DEK rotation re-encrypts a tenant's
  secrets), runnable without downtime — old prefix decrypts, new prefix writes.
- **Tests-first:** round-trip under the new scheme; cross-tenant isolation (tenant A's DEK can't
  decrypt tenant B's ciphertext); a rotation re-encrypts and the old key still reads pre-rotation rows
  until the pass completes.
- **Risk:** key management bugs lose data. Keep `SecretCipher` the only path; never store a DEK
  unwrapped; back up the KEK.

## v3.6c — Out-of-proc plugin sandboxing

The real isolation boundary, and the biggest item in the whole arc. Each plugin loads in its own
**process** with its own dependency closure, behind a serialized boundary — so a plugin can't read
another's memory, can't crash the host, and gets its own resource limits.

- **Seam:** replace the in-proc `PluginLoadContext` + reflection dispatch with a host↔plugin protocol
  (a thin RPC over stdio/pipes), the host marshalling action/trigger/connection calls. **This retires
  the in-proc workarounds** — `PluginLoadContext` name rules and `PluginReflection.LoadableTypes`
  partial-failure tolerance — *delete* them when it lands rather than carry them.
- **Hard parts:** the SDK boundary becomes a wire contract (versioned), config/result still flow as
  JSON (already the case), and trigger listeners that long-poll need lifecycle management across the
  process boundary. Hot-reload becomes process restart.
- **Tests-first:** an action runs out-of-proc and returns the same result; a crashing plugin fails its
  step without taking down the host; a plugin can't reach another plugin's or the host's internals.
- **Risk:** largest surface in v3.x. Consider shipping it as its own release (v3.7 / v4-preview) and
  doing a design spike first; if it slips, v3.6 is still a coherent release (audit + DEKs).
